## Real finding: ISR write storm from layout cascade

Even at $0.87/day, ISR Writes was still $0.45/day. User pulled Vercel Observability ISR-by-route data:

| Route | Writes | Unique Paths | Writes/Path |
|---|---|---|---|
| /live-feed-updates/[slug] | 19K | 13K | 1.46 |
| /bigbrother-players/[slug] | 8K | 366 | 22 |
| /[slug] | 6.1K | 1.9K | 3.2 |
| /index | 566 | 1 | 566 |
| /directory | 532 | 1 | 532 |
| /live-feed-updates | 529 | 1 | 529 |
| /become-supporter | 515 | 1 | 515 |
| /contact | 514 | 1 | 514 |
| /privacy-policy | 356 | 1 | 356 |
| /bigbrother-seasons/[slug] | 254 | 48 | 5.3 |
| /tag/[slug] | 74 | 31 | 2.4 |
| /category/[...slug] | 62 | 26 | 2.4 |
| /login | 62 | 1 | 62 |

The static pages with single paths and zero dynamic content rebuilding 350-566 times per day. Smoking gun: they all share the root layout, nothing else. Confirmed Next.js behavior: a layout fetch's revalidate floors every page in the segment to that timer.

Two layout fetches were the culprit:
1. `getAdScripts()` in `src/lib/api/ads.js:16` — `revalidate: 300` (5min floor)
2. Inline fetch for `/bbjd/v1/ad-settings` in `src/app/layout.jsx:98` — `revalidate: 3600` (1h floor, additional)

Round 1 (May 1) memo had explicitly flagged `getAdScripts()` as the static-page floor and deferred fixing it. Three rounds of fixes later, this was the leftover.

## The fix shipped (commit `671fa9d` on staging → `5ef3e04` on main, ~11am Pacific)

- `lib/api/ads.js` `getAdScripts`: `revalidate: 300` → `revalidate: false`
- `app/layout.jsx` inline ad-settings fetch: added `tags: ["ad-settings"]`, `revalidate: 3600` → `revalidate: false`
- `app/api/revalidate/route.js`: added `case "ad-scripts":` and `case "ad-settings":` for tag-based purge
- Updated layout comment to flag the layout-fetch landmine alongside the existing cookies/headers warning

WP-side trigger to fire `ad-settings`/`ad-scripts` webhooks on save NOT yet wired. Relies on manual purge via `POST /api/revalidate { secret, tag: "ad-scripts" }` until added. Ad scripts change quarterly so this is fine.

## Architectural rule (codified in memory)

**No time-based `revalidate` on layout fetches, ever.** Anything in a shared route segment must be `revalidate: false` + tag-based. The cascade is silent and expensive.

## In-season scaling Q&A

User asked how the architecture works in BB28 season. Answer:
- Pure webhook-driven now. Feed updates / posts / comments / players all fire webhooks → tags invalidated → next visitor causes 1 ISR write → cached until next webhook
- No page-level auto-revalidate timers anywhere
- Some 1h fetch-level timers remain inside specific pages (users.js, contact.js, posts list fetches, etc.) — these don't cascade because they're not in layouts
- Spoiler-bar webhook fans out to 6 pages (anything rendering `<SpoilerBarWrapper />`) — the only multi-page invalidation vector left
- Projected in-season writes: ~7K-18K/day (~1-2x current). Cost on writes: ~$0.05-0.10/day. Other lines (Fluid CPU, Origin Transfer) scale with traffic.
