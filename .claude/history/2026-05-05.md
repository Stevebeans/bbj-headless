## Live bug fixed: feed update pages 500ing (92% error rate)

Vercel observability showed `/live-feed-updates/[slug]` at 92% 5XX. Root cause:

1. Yesterday's siteurl fix flipped WP-generated URLs from `bigbrotherjunkies.com` → `wp.bigbrotherjunkies.com`
2. `next.config.js` `images.remotePatterns` had `bigbrotherjunkies.com` and `stg-wp.bigbrotherjunkies.com` but **not `wp.bigbrotherjunkies.com`**
3. Every uncached feed update page tried to render `<Image src="https://wp.bigbrotherjunkies.com/.../avatar.webp" />` → next/image rejected unconfigured host → render threw → 500
4. Pages already ISR-cached before the siteurl change kept serving (cached HTML had old URLs)

**Fix:** added `wp.bigbrotherjunkies.com` to `images.remotePatterns`. Note: `unoptimized: true` does NOT bypass remotePatterns hostname validation in Next.js 15.

Actually shipping the fix took several rounds because builds had been silently failing since Monday's recovery (more below).

## Cascade: Vercel builds failing since Monday

Discovered that no production deploy had succeeded since 2026-05-02. Every push since the recovery had been erroring. The site had been serving the May 2 build the whole time. Investigated 403s in build logs.

Chain of root causes (in order found):

1. **Initial guess (wrong):** `WORDPRESS_API_URL` env on Vercel pointed at apex. **Verified by user it didn't.** Already correctly set to `wp.bigbrotherjunkies.com/wp-json`.

2. **Stale CF edge cache.** Earlier today (Monday afternoon during the broken-auth period), Cloudflare's edge cached 30-day-TTL 403 responses (because Breeze was injecting `s-maxage=2592000` on all responses including auth-protected /wp-json/* — see yesterday's history). Different CF POPs serve different POPs to Vercel build region (iad1) vs the user's home POP — user's earlier surgical purges fixed her POP but Vercel's build POP still had stale 403s. **"Purge Everything" got most of them out**, but build still failed on different paths.

3. **The actual blocker: Cloudflare Bot Fight Mode.** It auto-blocks data-center IPs as bots, including Vercel's build agents. The mitigation rate on the bigbrotherjunkies.com zone was **31%** (extreme — normal is < 5%) — that's how much traffic CF was blocking/challenging.

   Resolved with a multi-step test:
   - Toggled both Bot Fight Mode + Block AI bots OFF → build succeeded
   - Toggled Block AI bots back ON → build still succeeded (so Block AI bots is fine to keep)
   - Conclusion: **Bot Fight Mode is the blocker; Block AI bots uses a different mechanism (UA-pattern matching) that doesn't catch Vercel**
   
   Permanent state: Bot Fight Mode OFF, Block AI bots ON, plus a WAF Skip rule on `wp.bigbrotherjunkies.com` covering all rate limiting / managed rules / Super Bot Fight Mode / User Agent Blocking / Browser Integrity Check / Security Level (the Skip rule alone wasn't catching the Bot Fight Mode managed rule, which is why we also had to toggle).

   In this architecture, Bot Fight Mode adds no real protection — apex is Vercel (which has its own DDoS protection), wp subdomain only sees server-to-server build traffic + admin browsing, and reCAPTCHA gates the forms. Once compare-page bot crawls are addressed (next item), there's no remaining vector that needs Bot Fight Mode.

## Structural fixes shipped (3 commits)

### 1. Compare page restructure (`a7efa39`)

Was: `/compare/[matchup]` with `revalidate = false` + `dynamicParams = true`. Each visited matchup combo (~40k possible across 200 players) triggered an ISR cache write on first hit. Bots could crawl the combinatorial space. Showed up in observability as compare-write-rate > read-rate.

Now: `/compare?p1=x&p2=y` with `searchParams`. Reading searchParams forces the page Dynamic — zero ISR writes per combo. Player data fetches still cached via the existing `getPlayerBySlug` fetch cache, so cost is bounded.

Also removed:
- `SuggestedPlayerComparisons` component on player profile pages (the "Compare X" block with 6 explicit `<a>` tags) — was the largest bot-discovery vector for compare URLs
- Updated all internal callers (`PlayerPicker`, `RelatedPlayers`, `SuggestedComparisons` on the compare page itself, `ComparisonJsonLd` canonical URL) to use `?p1=&p2=` format

Kept the small hover-icon compare button on the Castmates grid (low-prominence UX, already hover-only).

Net: 12 files changed, -140 lines. `/compare` is now `ƒ Dynamic` in build output.

### 2. Edit pages: kill timer-based ISR (`8625c44`)

`/bigbrother-players/[slug]/edit` and `/bigbrother-seasons/[slug]/edit` had `revalidate = 60` — every visited slug rebuilt every minute, all day, even though admins barely visit. Per observability: 609 invocations/12h on the players edit page alone.

Fix: removed `revalidate`, `dynamicParams`, `generateStaticParams` from both pages. Added both to the middleware matcher (cookies access forces dynamic, no ISR caching). Auth gating still happens client-side in the form components.

### 3. Legacy redirects + WP-probe middleware bail + roadmap fix (`173fb82`)

Six 308 redirects added to `next.config.js` for legacy WP slugs hitting the `/[slug]` catch-all and creating ISR writes:
- `/premium` → `/become-supporter` (1.1K/day — biggest win)
- `/log-in` → `/login` (82/day)
- `/feed-updates` → `/live-feed-updates` (149/day)
- `/player-directory` → `/directory` (75/day)
- `/registration` → `/login` (87/day)
- `/big-brother-stats` → `/directory` (74/day)

Middleware now early-bails with 404 on `/wp-login.php`, `/wp-admin/*`, `/xmlrpc.php` to skip function invocation on attack probes (38/day on /wp-login.php alone).

**Discovery while verifying:** Vercel itself returns `403 Forbidden` with `x-vercel-mitigated: deny` for these probe paths — its built-in mitigation already blocks them at the edge before middleware runs. So our middleware bail is redundant (but harmless). Could remove later for leanness; not urgent.

Roadmap (`/about` was marked done, actually 404s with 642 hits/day) corrected to `[ ]` with a note about needing creation.
