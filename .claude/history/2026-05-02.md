## Theme of the day
Ad system gating + cache hygiene + admin tools. Branch: `staging`.

## What got done

### 1. Caching-first rule, codified
- Added "CRITICAL: Caching Comes First — Always" section to `CLAUDE.md` (above "API Caching & Revalidation")
- Hard rules: never `cookies()`/`headers()` in shared layouts, never `force-dynamic` on content pages, narrow ISR tags, per-user data stays client-side
- New memory: `feedback_caching_first.md` — gatekeeper rule for all future bbj-app work

### 2. Freestar SDK supporter-fix + PSI defer
- Diagnosed: SDK was loading for ALL users (Pulte sticky / Toyota footer / mobile anchor) because `layout.jsx` rendered the `<Script>` based only on the anonymous-baseline `initialShouldShowAds`
- React 19 quirk: `<link rel="stylesheet">` hoists to `<head>` and persists; once Freestar loads it can't be unloaded
- **New file:** `src/components/ads/FreestarSDKLoader.jsx` — client component, renders SDK only when `!loading && shouldShowAds`
- **Modified:** `src/app/layout.jsx` — removed Freestar `<Script>`/`<link>` blocks, mounted `<FreestarSDKLoader />` inside `<Providers>`
- SDK strategy: `afterInteractive` → `lazyOnload` (TBT win for anonymous PSI runs)
- **Verified working in dev:** loader logs show `decision: "SKIP SDK"` for admin role
- Cache impact: zero — layout still 100% static / ISR-friendly
- Preconnects (8 of them, all Freestar-related) now only render after auth hydration, no longer block FCP

### 3. Spoiler bar audit
- User believed spoiler bar should only refresh when roster status page is saved → **confirmed correct**
- Tracing: `revalidateSpoilerBar()` fires from `SeasonRoutes.php:719` (`updateRosterStatus`) and `:778` (`purgeSeasonCache`) only
- Player profile edits / photo uploads correctly do NOT cascade to spoiler bar (intentional design)
- One real gap: add/remove player from season busts WP cache but does NOT call `revalidateSpoilerBar()`. User said leave for now.
- Recanted earlier "webhook gap" alarm about `case "player"` not invalidating spoiler-bar — that was correct by design

### 4. Admin Purge Cache button
- **New button on `/admin` overview** under new "Site Maintenance" section
- Flow: button → confirm dialog → `purgeCache()` → WP `POST /admin/purge-cache` → `Revalidation::revalidateAll()` → Next.js `/api/revalidate` `type: "all"` → `revalidatePath("/", "layout")` (Next 15 nuclear)
- Files changed:
  - `bigbrotherjunkies-data/src/Utils/Revalidation.php` — added `revalidateAll()`
  - `bigbrotherjunkies-data/src/Api/AdminRoutes.php` — added route + `purgeCache()` handler, gated by existing `checkAdminAccess`
  - `bbj-app/src/lib/api/admin.js` — added `purgeCache()` client fn
  - `bbj-app/src/app/admin/page.jsx` — button + confirm + loading/success/error states

### 5. Related posts 400 fix (defensive)
- WP returned 400 (`rest_invalid_param`) on `getRelatedPosts` — `categoryId` and/or `postId` were non-integer values getting interpolated into the URL
- Fixed `src/lib/api/posts.js` — `parseInt` + `Number.isFinite` check + `> 0` rejection before fetch
- Already caught/handled gracefully so wasn't user-facing, just log noise

### 6. Smaller stuff
