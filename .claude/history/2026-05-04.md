## Afternoon: post-recovery debugging — admin/settings broken

After the morning rewire, `/admin` showed "Access Denied" and `/settings` showed "Failed to load settings". Multi-layer root cause that took unwinding:

### Root cause chain

1. **wp-cli broken on new server** — `wp-config.php` has `require "wp-salt.php";` (relative path). wp-cli's runner can't resolve relative requires from its eval'd context. Every `wp option update` from the morning rewire silently no-op'd. Need to change to `require __DIR__ . "/wp-salt.php";` to fix wp-cli.

2. **Production siteurl was set to staging URL** — because the morning's `wp option update siteurl` never executed (point 1), the recovered DB kept `https://stg-wp.bigbrotherjunkies.com` in `wp_options.siteurl` and `home`. When users logged in fresh, the BBJ plugin's `AuthRoutes.php:1316` signed JWTs with `iss = get_bloginfo('url')` = staging URL. The standard jwt-auth plugin verified `$token->iss !== get_bloginfo('url')` (also staging URL) — should match, but something further downstream rejected. Updated via direct PHP eval against `wp-load.php` + Redis FLUSHALL.

3. **Breeze plugin was injecting `s-maxage=2592000` on EVERY response** — including `/wp-json/*` auth endpoints. Cloudways' default cache plugin (Breeze) writes this `.htaccess` block on activation:
   ```apache
   <IfModule mod_headers.c>
      Header append Cache-Control "s-maxage=2592000"
   </IfModule>
   ```
   With no path conditional. Result: Cloudflare cached 30 days of error responses (and would have cached cross-user authenticated JSON). Deactivated Breeze entirely — it provides nothing in our Vercel-fronted stack and only adds risk. `auto-cache-purge.php` mu-plugin is safe (uses `function_exists('breeze_flush_cache')`).
